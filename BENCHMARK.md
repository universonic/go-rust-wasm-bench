# Benchmark Design: Go vs Rust in WebAssembly Application Development

## 1. Overview

This document describes the benchmark suite used to compare **Go** (standard compiler and TinyGo) with **Rust** for WebAssembly (Wasm) application development. The suite covers two deployment scenarios — **browser-side** and **WASI-based microservice** — with two test cases each, for a total of **four test cases**. Every test case is grounded in empirical research on real-world Wasm workloads and is designed to be simple, rigorous, and reproducible.

### 1.1 Toolchain Matrix

Each test case is compiled by three toolchains, producing three `.wasm` artifacts:

| Toolchain | Browser Target | WASI Target |
|-----------|---------------|-------------|
| **Go (standard)** | `GOOS=js GOARCH=wasm` | `GOOS=wasip1 GOARCH=wasm` |
| **TinyGo** | `tinygo build -target=wasm` | `tinygo build -target=wasip1` |
| **Rust** | `wasm32-unknown-unknown` + wasm-bindgen | `wasm32-wasip1` |

### 1.2 Test Case Summary

| ID | Test Case | Scenario | Workload Type |
|----|-----------|----------|---------------|
| B1 | Image Convolution | Browser | Compute-intensive + memory-access-intensive |
| B2 | JSON Round-trip | Browser | Data processing + JS-Wasm interop |
| B3 | SHA-256 Hashing | Microservice (WASI) | Pure compute-intensive (integer) |
| B4 | JSON Round-trip | Microservice (WASI) | Data serialization / deserialization |

## 2. Test Cases

### 2.1 B1 — Image Convolution (Browser)

#### 2.1.1 Covered Scenarios

Image convolution represents the **compute-intensive pixel-level processing** pattern that underlies a significant portion of browser-side Wasm workloads:

- **Media processing / Player** (9% of web Wasm binaries) — direct match: pixel-level filtering and transformation are core operations in media processing applications.
- **Visualization / Animation** (11%) — rendering filters and visual effects use the same sliding-window dense arithmetic pattern.
- **Games** (25%) — partially covered: game engines use similar dense numerical computations for physics, particle effects, and post-processing.

Production examples include Figma (C++ rendering engine compiled to Wasm, 3x load time reduction [R1]), Adobe Photoshop Web (image processing in Wasm), and PSPDFKit (PDF rendering via Wasm).

**Estimated coverage**: ~25–35% of browser-side Wasm use cases share the same core computational pattern.

#### 2.1.2 Design Rationale

Image convolution is chosen over simpler alternatives (e.g., grayscale conversion) because:

1. It exercises both **compute intensity** (multiply-accumulate operations per pixel) and **memory access locality** (sliding-window pattern), which are the two dominant performance factors in Wasm execution [R2].
2. A peer-reviewed study (SPIE 2023) directly validated image convolution as a representative Wasm benchmark, finding Wasm achieves significant speedups over JavaScript for this workload [R3].
3. By testing two kernel sizes (3×3 and 5×5), we can observe how performance scales with arithmetic intensity — a key factor for understanding compiler optimization effectiveness.

#### 2.1.3 Specification

**Input**:
- RGBA pixel buffer (4 bytes per pixel), with pixel values generated by a PRNG seeded with **fixed seed 42**.
- Image dimensions (width × height): `256×256`, `512×512`, `1024×1024`, `1920×1080`.

**Operation**:
- Apply 2D convolution with the following normalized Gaussian kernels:

  3×3 Gaussian kernel (σ ≈ 0.85):

  ```
  K3 = (1/16) * [[1, 2, 1],
                  [2, 4, 2],
                  [1, 2, 1]]
  ```

  5×5 Gaussian kernel (σ ≈ 1.0):

  ```
  K5 = (1/256) * [[1,  4,  6,  4, 1],
                   [4, 16, 24, 16, 4],
                   [6, 24, 36, 24, 6],
                   [4, 16, 24, 16, 4],
                   [1,  4,  6,  4, 1]]
  ```

- Convolution is applied **per-channel** (R, G, B independently; A is copied unchanged).
- Border handling: **zero-padding** (out-of-bounds pixels treated as 0).
- Output pixel values are clamped to `[0, 255]` and rounded to the nearest integer.

**Output**: RGBA pixel buffer of the same dimensions as input.

**Correctness**: Output buffer is verified against a precomputed reference (per-pixel tolerance: ±1 due to rounding).

### 2.2 B2 — JSON Round-trip (Browser)

#### 2.2.1 Covered Scenarios

JSON round-trip in the browser tests two critical dimensions:

- **JS-Wasm data exchange**: In real-world browser Wasm applications, structured data must cross the JS-Wasm boundary. This crossing involves memory copying, string encoding/decoding (UTF-8 ↔ UTF-16), and buffer management — a known performance bottleneck [R4].
- **Dynamic memory allocation patterns**: JSON parsing requires frequent allocations and deallocations, which is exactly where the GC-based (Go) vs ownership-based (Rust) memory management models exhibit the most significant differences.
- **Text processing** (11% of web Wasm binaries): The most widely deployed Wasm binary on the web is Hyphenopoly (a text processing library, present on 34.9% of Wasm-using domains) [R5]. JSON parsing exercises the same string processing and memory allocation patterns.

**Estimated coverage**: ~15–20% of browser-side Wasm use cases directly; the JS-Wasm data exchange pattern has broader applicability to virtually all browser Wasm applications that interact with JavaScript.

#### 2.2.2 Design Rationale

1. JSON is the lingua franca of web data exchange. Any browser Wasm module that receives input from or returns results to JavaScript will likely handle JSON-like structured data.
2. The operation includes **deserialization + business logic (filter & sort) + serialization**, representing a realistic end-to-end data processing pipeline rather than a synthetic micro-benchmark.
3. Using the **same data and logic** as B4 (WASI scenario) enables direct cross-scenario comparison.

#### 2.2.3 Specification

**Input**: A JSON string containing an array of `N` user records. Each record has the following schema:

```json
{
  "id": <int>,
  "name": "<string>",
  "email": "<string>",
  "age": <int>,
  "score": <float64>
}
```

Records are generated deterministically: `id` is sequential starting from 1; `name`, `email`, `age`, and `score` are derived from a PRNG with **fixed seed 42**.

- `name`: 8–16 random lowercase ASCII characters.
- `email`: `<name>@example.com`.
- `age`: random integer in `[10, 80]`.
- `score`: random float in `[0.0, 100.0]`, rounded to 2 decimal places.

**Sizes**: N = `100`, `1,000`, `10,000`.

**Operation**:
1. **Deserialize** the JSON string into an array of typed structs/objects.
2. **Filter**: retain only records where `age >= 18`.
3. **Sort**: order the remaining records by `score` in **descending** order. When scores are equal, order by `id` in ascending order (to ensure deterministic output).
4. **Serialize** the resulting array back to a JSON string.

**Output**: JSON string.

**Correctness**: Output JSON is verified against a precomputed reference via semantic comparison (parsed data structures must be equal; see Section 4.4 for details on why byte-exact matching is insufficient).

### 2.3 B3 — SHA-256 Hashing (Microservice / WASI)

#### 2.3.1 Covered Scenarios

SHA-256 hashing represents the **pure compute-intensive integer workload** pattern that is routine in microservice / serverless environments:

- **Authentication and signing**: API gateways, JWT validation, HMAC computation.
- **Data integrity**: Content-addressable storage, cache validation, checksum verification.
- **Cryptographic operations**: A 2025 comparative study confirmed that cryptographic operations are a well-documented and representative Wasm workload category [R6].

In the WASI/serverless context, pure compute functions are where Wasm exhibits its greatest advantages over containers: up to 30× smaller images and 16% lower cold-start latency [R7].

**Estimated coverage**: ~30–40% of compute-intensive microservice Wasm functions.

#### 2.3.2 Design Rationale

1. SHA-256 is chosen over other hash functions because it is universally available in both Go's standard library (`crypto/sha256`) and Rust's ecosystem (`sha2` crate), ensuring a fair comparison using idiomatic, production-quality implementations.
2. When compiled to Wasm, neither Go nor Rust can use CPU-specific cryptographic instruction set extensions (e.g., SHA-NI, AES-NI), so both fall back to pure software implementations — making the comparison fair at the algorithmic and compiler optimization level.
3. SHA-256 is **integer-heavy** (32-bit additions, rotations, bitwise operations), complementing B1's float-heavy workload and B2/B4's allocation-heavy workload.

#### 2.3.3 Specification

**Input**: A byte buffer of size `S`, filled with a repeating pattern `0x00, 0x01, 0x02, ..., 0xFF, 0x00, 0x01, ...` (deterministic, no PRNG needed).

**Sizes**: S = `1 KB` (1,024 bytes), `64 KB` (65,536 bytes), `1 MB` (1,048,576 bytes), `16 MB` (16,777,216 bytes).

**Operation**: Compute the SHA-256 digest of the input buffer.

**Implementation**: Use each language's standard/canonical library:
- Go: `crypto/sha256`
- TinyGo: `crypto/sha256` (same API, TinyGo-compatible subset)
- Rust: `sha2` crate (`sha2::Sha256`)

**Output**: 32-byte (256-bit) digest, hex-encoded.

**Correctness**: Output digest is verified against a precomputed reference for each input size. Reference values can be independently verified using `sha256sum` on the input files.

### 2.4 B4 — JSON Round-trip (Microservice / WASI)

#### 2.4.1 Covered Scenarios

JSON serialization/deserialization is the **dominant data exchange bottleneck** in Wasm microservices:

- The Roadrunner project (arXiv, 2024) found that **97% of data transfers** in Wasm serverless workflows involve serialization overhead, and optimizing this single aspect achieved 44–89% latency reduction and 69× throughput improvement [R8].
- The Lumos benchmarking study (ACM IoT 2025) identified I/O serialization as causing up to 10× overhead in interpreted Wasm and up to 2× in AOT-compiled Wasm [R7].

**Estimated coverage**: ~70%+ of microservice Wasm data processing patterns (since nearly all serverless functions involve JSON input/output).

#### 2.4.2 Design Rationale

1. JSON processing is the single most representative microservice workload based on the research evidence.
2. Using **identical data, schema, and operation logic** as B2 (browser scenario) enables a unique **cross-scenario comparison**: the same computation running in two different Wasm environments (browser JS engine vs Wasmtime).
3. In the WASI context, there is no JS-Wasm interop overhead; data is passed via **stdin** and results via **stdout**, isolating the pure JSON processing performance.

#### 2.4.3 Specification

**Identical to B2** (Section 2.2.3) with the following differences:

- **Input delivery**: JSON string is read from **stdin** (as a WASI file descriptor).
- **Output delivery**: Result JSON string is written to **stdout**.
- **No JS-Wasm interop**: Data stays within the Wasm linear memory throughout processing.

## 3. Metrics

### 3.1 Compilation Metrics

| ID | Metric | Method | Unit |
|----|--------|--------|------|
| C1 | Binary size | Size of the `.wasm` file after compilation with standard optimization flags | Bytes |
| C2 | Build time | Median of 5 consecutive **clean builds** (caches cleared before each build) | Seconds |

**Optimization flags**:
- Go: `-ldflags="-s -w"` (strip symbol table and debug info)
- TinyGo: `-opt=z -no-debug`
- Rust: `--release` with `[profile.release] opt-level = "z"`, `lto = true`, `strip = true`, `codegen-units = 1`

### 3.2 Browser Runtime Metrics

| ID | Metric | Method | Unit |
|----|--------|--------|------|
| R1 | Module instantiation time | Wall-clock time from `WebAssembly.instantiateStreaming()` call to the resolved promise | ms |
| R2 | Execution time | `performance.now()` around the kernel call; mean of **30 iterations** after **5 warmup** iterations | ms |
| R3 | Memory delta | `performance.measureUserAgentSpecificMemory()` sampled before first run and after last run; report the delta | Bytes |

### 3.3 WASI Runtime Metrics

| ID | Metric | Method | Unit |
|----|--------|--------|------|
| R4 | Cold-start time | Full `wasmtime run` invocation (process start to exit); measured by `hyperfine --warmup 3 --runs 30` | ms |
| R5 | Execution time | Internal timing within the Wasm module (clock_gettime or equivalent); mean of **30 iterations** after **5 warmup** iterations | ms |
| R6 | Peak memory | Max resident set size reported by `/usr/bin/time -l` (macOS) or `/usr/bin/time -v` (Linux) | Bytes |

### 3.4 Engineering Metrics

| ID | Metric | Method | Unit |
|----|--------|--------|------|
| E1 | Source lines of code | Measured by `cloc`, excluding comments and blank lines; count only the benchmark implementation files | Lines |
| E2 | Toolchain complexity | Qualitative assessment of build steps, required configuration, and dependency management | — |

## 4. Test Procedure

### 4.1 Environment Requirements

- **Hardware**: All benchmarks must run on the **same physical machine**. Document: CPU model, core count, clock speed, RAM size.
- **OS**: macOS or Linux. Document: exact version.
- **Browser**: Google Chrome (latest stable). Document: exact version and V8 version.
- **WASI runtime**: Wasmtime (latest stable). Document: exact version.
- **Compilers**: Document exact versions of Go, TinyGo, Rust (rustc), wasm-bindgen, wasm-pack.

### 4.2 Browser Scenario Procedure

```
For each toolchain T in {Go, TinyGo, Rust}:
  For each test case B in {B1, B2}:
    1. Clean build: remove all build artifacts
    2. Compile B with T targeting browser Wasm
    3. Record C1 (binary size) and C2 (build time, repeat 5×, take median)
    4. Deploy the .wasm file with a minimal HTML test harness
    5. Start a local HTTP server (Node.js, with COOP/COEP headers for memory API)
    6. Launch Chromium via Playwright (headless mode, consistent flags)
    7. For each input size S:
       a. Load the Wasm module → record R1 (instantiation time)
       b. Run 5 warmup iterations (discard results)
       c. Run 30 measurement iterations → record R2 (execution time per iteration)
       d. Record R3 (memory delta)
       e. Capture the output and verify correctness against reference
    8. Export all metrics to JSON
```

**Chrome launch flags** (for consistency):
```
--disable-extensions --disable-background-networking
--disable-default-apps --no-first-run
--disable-gpu --js-flags="--no-opt"
```

Note: `--js-flags="--no-opt"` is **not** applied to the Wasm execution itself (Wasm is always compiled by V8's TurboFan/Liftoff); it prevents JavaScript JIT optimizations from interfering with timing measurements of the JS glue code.

### 4.3 WASI / Microservice Scenario Procedure

```
For each toolchain T in {Go, TinyGo, Rust}:
  For each test case B in {B3, B4}:
    1. Clean build: remove all build artifacts
    2. Compile B with T targeting wasip1
    3. Record C1 (binary size) and C2 (build time, repeat 5×, take median)
    4. For each input size S:
       a. Generate the deterministic test input file (shared across all toolchains)
       b. Cold-start measurement:
          hyperfine --warmup 3 --runs 30 \
            'wasmtime run module.wasm < input_S.bin'
          → record R4
       c. Warm execution measurement:
          wasmtime run module.wasm --bench 35 < input_S.bin
          The module internally runs 5 warmup + 30 measured iterations,
          prints per-iteration timing to stderr (stdout carries the output)
          → record R5
       d. Peak memory measurement:
          /usr/bin/time -l wasmtime run module.wasm < input_S.bin
          → record R6 (max resident set size)
       e. Capture output and verify correctness against reference
    5. Export all metrics to JSON
```

### 4.4 Correctness Verification

Before any performance measurement, **all implementations must produce identical outputs** for all input sizes:

1. Generate reference outputs using a standalone generator program (`testdata/generate.go`) that deterministically produces all inputs and reference outputs.
2. For each (toolchain × test case × input size), compare the Wasm output against the reference:
   - B1: per-pixel comparison with tolerance ±1 (due to floating-point rounding).
   - B2/B4: **semantic JSON comparison** — parse both outputs and compare the resulting data structures. This accounts for serialization differences between Go and Rust (e.g., Go's `encoding/json` serializes `95.0` as `95` while Rust's `serde_json` preserves `95.0`).
   - B3: exact hex digest match.
3. Any mismatch is a **blocking error** — the implementation must be fixed before proceeding with performance measurements.

## 5. Statistical Analysis

### 5.1 Reporting

For each metric, report:
- **Mean**, **median**, **standard deviation**, **min**, **max**.
- Number of samples: N = 30 for runtime metrics, N = 5 for build time.

### 5.2 Visualization

- **Box plots**: execution time distributions (R2, R4, R5) across toolchains.
- **Bar charts**: binary sizes (C1) and build times (C2).
- **Line charts**: execution time vs. input size, to show scaling behavior.
- **Scatter plots**: cross-scenario comparison (browser vs. WASI) for identical workloads.
- **Radar chart**: multi-dimensional overview across all key metrics (C1, C2, R1–R6, E1, E2) with min-max normalization.

### 5.3 Significance Testing

- Use the **Mann-Whitney U test** (non-parametric, does not assume normal distribution) to determine whether performance differences between toolchains are statistically significant.
- Significance levels: α = 0.05 (*), α = 0.01 (**), α = 0.001 (***).
- Report p-values for each pairwise comparison (Go vs Rust, TinyGo vs Rust, Go vs TinyGo).
